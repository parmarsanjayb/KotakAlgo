# SPV Quantum AI - AI Agent Framework

This document outlines the design, execution sequence, communication flows, and telemetry logging configurations for the reusable **AI Agent Framework** within SPV Quantum AI.

---

## 1. Class Diagram

The framework relies on a common `BaseAgent` abstract class and a structured `BaseEvent` model class.

```mermaid
classDiagram
    class BaseEvent {
        +event_id: str
        +timestamp: datetime
        +source_agent: str
        +priority: EventPriority
        +topic: str
        +payload: dict
    }
    
    class MarketDataEvent {
        +topic: "market_data"
    }
    class RiskEvent {
        +topic: "risk_alerts"
    }
    class SignalEvent {
        +topic: "trade_signals"
    }
    class ExecutionEvent {
        +topic: "execution_reports"
    }

    BaseEvent <|-- MarketDataEvent
    BaseEvent <|-- RiskEvent
    BaseEvent <|-- SignalEvent
    BaseEvent <|-- ExecutionEvent

    class BaseAgent {
        <<abstract>>
        +id: str
        +name: str
        +description: str
        +status: str
        +confidence_score: float
        +health_status: str
        +execution_time: float
        +logs: List[str]
        +last_decision: dict
        +input_events: List[str]*
        +output_events: List[str]*
        +initialize()
        +shutdown()
        +start()
        +stop()
        +receive_event(event: BaseEvent)
        +analyze(event: BaseEvent)* dict
        +publish_result(topic, payload)
    }

    class MarketAgent {
        +input_events: ["market_control"]
        +output_events: ["market_data"]
        +on_tick()
        +analyze()
    }
    class RiskAgent {
        +input_events: ["order_request", "portfolio_update"]
        +output_events: ["risk_alert", "order_rejected", "order_approved"]
        +analyze()
    }
    class ExecutionAgent {
        +input_events: ["order_approved"]
        +output_events: ["order_filled", "portfolio_update", "execution_failed"]
        +analyze()
    }

    BaseAgent <|-- MarketAgent
    BaseAgent <|-- RiskAgent
    BaseAgent <|-- ExecutionAgent

    class EventBus {
        -_subscribers: dict
        +subscribe(topic, callback)
        +publish(topic, sender, data)
    }

    class AgentRegistry {
        -_registered_classes: dict
        +discover_agents()
        +get_agent_class()
    }

    class AgentManager {
        +active_agents: dict
        +load_agents()
        +start_all()
        +stop_all()
        -_monitor_loop()
        +audit_health_and_supervise()
    }

    AgentManager --> BaseAgent : manages
    AgentRegistry --> BaseAgent : discovers
    BaseAgent --> EventBus : publishes/subscribes
```

## 2. Sequence Diagram

The sequence below traces the event flows during a trade order evaluation and execution.

```mermaid
sequenceDiagram
    autonumber
    actor User as Web Client
    participant Bus as EventBus
    participant Risk as RiskAgent
    participant Exec as ExecutionAgent
    participant Broker as PaperBroker
    participant DB as PostgreSQL
    participant Telegram as TelegramNotifier

    User->>Bus: publish("order_request", payload)
    Note over Bus: Event mapped to ExecutionEvent
    Bus->>Risk: receive_event(event)
    activate Risk
    Note over Risk: Middleware measures CPU/Memory
    Risk->>Risk: analyze(order_request)
    Note over Risk: Checks max positions & drawdown
    alt Passes Risk checks
        Risk->>Bus: publish_result("order_approved", payload)
        deactivate Risk
        Bus->>Exec: receive_event(event)
        activate Exec
        Exec->>Exec: analyze(order_approved)
        Exec->>Broker: place_order()
        activate Broker
        Broker->>DB: Write Order & Trade
        Broker-->>Exec: Return Order status (FILLED)
        deactivate Broker
        Exec->>Bus: publish_result("order_filled", fill_payload)
        Exec->>Bus: publish_result("portfolio_update", metrics)
        deactivate Exec
        Bus->>Telegram: receive_event(order_filled)
        Telegram->>Telegram: send_message()
    else Fails Risk checks
        Risk->>Bus: publish_result("order_rejected", error_payload)
        Risk->>Bus: publish_result("risk_alert", warning_payload)
        Bus->>Telegram: receive_event(risk_alert)
        Telegram->>Telegram: send_message()
    end
```

## 3. Communication Flow

All messages are represented as JSON strings. Every event generated by an agent conform to:

```json
{
  "event_id": "9b1deb4d-3b7d-4bad-9bdd-2b0d7b3dcb6d",
  "timestamp": "2026-07-04T05:15:30.123456+00:00",
  "source_agent": "risk_agent",
  "priority": "HIGH",
  "topic": "risk_alerts",
  "payload": {
    "alert_type": "LIMIT_EXCEEDED",
    "message": "Estimated cost $12000.00 exceeds limit $10000.00",
    "order_details": {
      "symbol": "BTCUSD",
      "side": "BUY",
      "quantity": 0.2,
      "price": 60000.0
    }
  }
}
```

## 4. API Design

### `GET /api/agent_stats`
Returns detailed running metrics for all instantiated agents under the supervisor manager.

#### Sample Response Payload:
```json
{

    "health": "HEALTHY",
    "confidence_score": 100.0,
    "execution_time_ms": 1.23,
    "last_decision": {
```
